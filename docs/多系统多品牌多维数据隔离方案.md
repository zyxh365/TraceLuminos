# TSP 可观测性平台 - 多系统多品牌多维数据隔离方案

> 版本：v1.0 | 更新日期：2026-04-21

---

## 一、背景

全链路监控平台需要接入多个 TSP 系统，每个 TSP 系统下又有多个品牌（OEM 客户）。需要在数据存储和查询层面支持：

- 按 **TSP 系统** 维度隔离（不同 TSP 实例的数据互不可见）
- 按 **品牌** 维度隔离（同一 TSP 系统下不同品牌的数据可按需查看）
- 按 **环境** 维度隔离（生产 / 测试 / 开发环境）
- 按 **渠道** 维度查看（奇瑞APP、捷途APP等不同业务渠道）
- 按 **平台** 维度查看（Android / iOS / H5 / Web / TBox，已有 `biz_platform` 字段）
- **租户** 维度预留扩展

### 维度层级

```
全链路监控平台
  └── TSP 系统（biz_system）
        └── 环境（env）
              └── 品牌（biz_brand）【预留】
                    └── 租户（biz_tenant）【预留】
                          └── 渠道/APP产品（biz_channel）
                                └── 技术平台（biz_platform）【已有】
                                      └── VIN（vin）
```

## 二、字段定义

| 字段 | 含义 | 数据类型 | 举例 | 状态 |
|------|------|---------|------|------|
| `biz_system` | TSP 系统标识 | LowCardinality(String) | `NTSP`、`STSP` | 新增 |
| `env` | 部署环境 | LowCardinality(String) | `production`、`test`、`dev` | 新增（替代 `deploy_env`） |
| `biz_channel` | 业务渠道/APP产品 | LowCardinality(String) | `chery-app`、`jetour-app`、`tata-app` | 新增 |
| `biz_platform` | 技术平台/终端类型 | LowCardinality(String) | `android`、`ios`、`h5`、`web`、`tbox` | 已有 |
| `biz_brand` | 品牌标识 | LowCardinality(String) | `chery`、`jetour`、`icar` | 新增（预留） |
| `biz_tenant` | 租户标识 | LowCardinality(String) | `lionaitech`、`oem-partner` | 新增（预留） |
| `vin` | VIN 码 | String | `LSVAU2A35N...` | 已有（`biz_vin`） |

> **兼容说明**：原有的 `deploy_env`、`biz_platform`、`biz_tenant_id`、`biz_vin` 字段保留不动，新字段与旧字段并存。新数据优先写入新字段，查询时可同时支持新旧两种过滤条件。

## 三、核心设计原则

### 3.1 注入分层

| 注入层级 | 负责注入的字段 | 注入方式 | 说明 |
|---------|--------------|---------|------|
| **APP 端（源头）** | `biz_channel`、`biz_brand`、`biz_tenant`、`vin` | OTel SDK Resource Attributes | 登录后从用户信息获取 |
| **HTTP 调用链路** | `biz_channel`、`biz_brand`、`biz_tenant`、`vin` | W3C Baggage 自动透传 | Agent / SDK 自动处理 |
| **Edge Collector** | `biz_system`、`env` | transform processor 配置写死 | 按部署实例确定 |

### 3.2 透传机制

业务维度字段（channel、brand、tenant、vin）采用 **W3C Baggage** 标准透传：

```
请求头：
baggage: biz.channel=chery-app,biz.brand=chery,biz.tenant=tenant-001,vin=LSVAU2A35N...
```

- APP 发起请求时，OTel SDK 自动将 Resource Attributes 写入 `baggage` header
- Kong 网关、Java 服务、Kafka 消费者均通过 OTel Agent/SDK 自动读取和透传
- **后端 Java 服务不需要写任何代码**

## 四、各端配置详解

### 4.1 APP 端（源头注入）

#### 4.1.1 Edge Collector 上报地址获取

APP 无法在内网直连 Edge Collector，需要通过公网域名中转。上报地址不应在 APP 中写死，而是由服务端下发。

**方式一：复用业务接口（推荐）**

在 APP 登录或初始化接口中顺带返回监控配置，不需要额外接口：

```json
// POST /api/app/init 或 GET /api/app/config
{
  "code": 200,
  "data": {
    "userInfo": { ... },
    "featureFlags": { ... },
    "monitorConfig": {
      "otelEndpoint": "https://otel-edge.tsp.com:4318",
      "bizChannel": "chery-app",
      "bizBrand": "chery"
    }
  }
}
```

**方式二：独立配置接口**

单独提供一个轻量配置接口，APP 启动时调用一次，与业务解耦：

```
GET /api/monitor/config
```

返回：

```json
{
  "otelEnabled": true,
  "otelEndpoint": "https://otel-edge.tsp.com:4318",
  "sampleRate": 0.1,
  "bizChannel": "chery-app",
  "bizBrand": "chery"
}
```

**服务端地址管理**

服务端本身不应写死地址，通过配置中心或环境变量管理：

```yaml
# application.yml
monitor:
  otel:
    endpoint: ${OTEL_EDGE_ENDPOINT:http://edge-collector:4318}
    enabled: ${OTEL_ENABLED:true}
    sample-rate: ${OTEL_SAMPLE_RATE:0.1}
```

```
生产环境：OTEL_EDGE_ENDPOINT=https://otel-edge-prod.tsp.com
测试环境：OTEL_EDGE_ENDPOINT=https://otel-edge-test.tsp.com
```

**公网入口架构**

```
APP（用户手机）
  → https://otel-edge.tsp.com:4318（公网域名）
    → Nginx / SLB 反向代理
      → Edge Collector（内网）
```

不同 TSP 系统的 APP 可通过子域名或 Header 路由到对应的 Edge Collector。

#### 4.1.2 OTel SDK Resource 初始化

APP 从服务端获取配置后，初始化 OTel SDK 的 Resource Attributes：

#### Android 端

```kotlin
// 登录成功后配置 OTel SDK
val resource = DefaultResource.builder()
    .put("service.name", "tsp-android-app")
    .put("biz.channel", "chery-app")
    .put("biz.brand", userInfo.brand)         // 从登录接口获取
    .put("biz.tenant", userInfo.tenantId)     // 从登录接口获取
    .put("vin", vehicleInfo.vin)              // 操作车辆时获取
    .build()

val sdk = OpenTelemetrySdk.builder()
    .setResource(resource)
    .setTracerProvider(tracerProvider)
    .build()
```

#### iOS 端

```swift
// 登录成功后配置 OTel SDK
let resource = Resource(attributes: [
    "service.name": AttributeValue.string("tsp-ios-app"),
    "biz.channel": AttributeValue.string("chery-app"),
    "biz.brand": AttributeValue.string(userInfo.brand),
    "biz.tenant": AttributeValue.string(userInfo.tenantId),
    "vin": AttributeValue.string(vehicleInfo.vin),
])
```

#### H5 / Web 端

```javascript
import { Resource } from '@opentelemetry/resources';

// 登录成功后配置
const resource = new Resource({
  ['service.name']: 'tsp-h5-app',
  ['biz.channel']: 'chery-app',
  ['biz.brand']: userInfo.brand,
  ['biz.tenant']: userInfo.tenantId,
  ['vin']: vehicleInfo.vin,
});
```

### 4.2 Kong 网关（透传）

Kong 通过 [opentelemetry 插件](https://docs.konghq.com/hub/kong-inc/opentelemetry/) 生成网关 Span，同时**自动透传**上游传来的 Baggage header：

```yaml
# Kong 配置（kong.yaml）
plugins:
  - name: opentelemetry
    config:
      endpoint: http://edge-collector:4318
      batch_count: 200
      batch_flush_delay: 2
```

- Kong 自身生成的 Span 携带上游 Baggage 中的业务维度字段
- 向下游 Java 服务转发请求时，Baggage header 自动透传
- **零额外配置**

### 4.3 Java 服务（自动透传）

OTel Java Agent 自动处理 Baggage 的读取和写入，**零代码**：

```yaml
# otel-javaagent.properties
otel.service.name=my-service
otel.exporter.otlp.endpoint=http://edge-collector:4317

# 开启 Baggage → Span Attributes 自动映射
otel.instrumentation.common.baggage-to-resource.enabled=true
```

开启后，Agent 自动将 Baggage 中的 `biz.*` 字段写入每个 Span 的 Attributes。

### 4.4 Edge Collector（补充基础设施维度）

每个 TSP 系统/环境部署独立的 Edge Collector，通过 transform processor 注入 `biz_system` 和 `env`：

```yaml
# edge-collector.yaml（示例：TSP 云平台 · 生产环境）
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  transform/traces:
    trace_statements:
      - context: resource
        statements:
          # 基础设施维度：配置写死，不可由上游覆盖
          - set(attributes["biz.system"], "tsp-cloud") where attributes["biz.system"] == nil
          - set(attributes["env"], "production") where attributes["env"] == nil

  batch:
    send_batch_size: 8192
    timeout: 5s

exporters:
  kafka:
    brokers: ["kafka:9092"]
    topic: "otel-traces"
    encoding: otlp_json

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [transform/traces, batch]
      exporters: [kafka]
```

> **不同 TSP 系统 / 环境只需修改 `biz.system` 和 `env` 的值**，其余配置完全一致。

## 六、部署架构

### 6.1 Collector 部署模式

按 **环境** 部署 Edge Collector，不按系统：

| 环境 | Edge Collector 数量 | `env` 值 | `biz.system` | 数据去向 |
|------|-------------------|---------|-------------|---------|
| 生产 K8s | 2（HA） | `production` | 按 Collector 实例写死 | 共享 Kafka |
| 测试 K8s | 1 | `test` | 按 Collector 实例写死 | 共享 Kafka |
| 开发 K8s | 1 | `dev` | 按 Collector 实例写死 | 共享 Kafka |

> 同一环境内，如果需要区分不同 TSP 系统，部署不同的 Edge Collector 实例，配置不同的 `biz.system` 值。

### 6.2 全局共享组件

- **Kafka**：1 套（所有 Edge Collector 写入）
- **Central Collector**：1 套（消费 Kafka，写入 ClickHouse）
- **ClickHouse**：1 套（存储全量数据，通过 `biz_system` / `env` 字段隔离）

## 七、新增一套 TSP 系统的接入流程

1. **部署 Edge Collector**：复制现有 Collector 配置，修改 `biz.system` 值
2. **各端配置上报地址**：将 OTel endpoint 指向新的 Edge Collector
3. **APP 端配置 Resource**：SDK 中设置 `biz.channel`、`biz.brand` 等
4. **ClickHouse**：无需任何改动（`biz_system` 字段已就绪）
5. **Kafka / Central Collector**：无需改动
6. **前端看板**：查询时增加 `biz_system` 过滤条件

详见 `scripts/clickhouse-schema/05_multi_dimension_schema.sql`，主要变更：

### 8.1 新增字段

涉及表：`tsp_spans`、`tsp_span_events`、`tsp_span_metrics`、`tsp_errors`、`tsp_service_topology`、`tsp_alert_rules`、`tsp_alert_events`

| 新增字段 | 类型 | 说明 |
|---------|------|------|
| `biz_system` | LowCardinality(String) | TSP 系统标识 |

| `biz_brand` | LowCardinality(String) | 品牌（预留） |
| `biz_tenant` | LowCardinality(String) | 租户（预留） |

### 8.2 索引

所有新增字段在 `tsp_spans` 和 `tsp_span_events` 上创建 bloom_filter 索引。

### 8.3 物化视图

- `mv_otel_to_spans`：增加 `biz.system`、`env`、`biz.channel`、`biz.brand`、`biz.tenant` 的 Resource/Span Attributes 映射
- `mv_otel_to_span_events`：同步增加多维字段映射
- `mv_errors`：重建，增加多维字段
- `mv_spans_hourly`：重建，增加多维字段到聚合维度
- `mv_service_topology_hourly`：重建，增加 `biz_system` 到聚合维度

### 8.4 视图

- `v_trace_detail`：增加多维字段输出
- `v_trace_summary`：增加多维字段聚合

### 8.5 表重建

`SummingMergeTree` 表（`tsp_span_metrics`、`tsp_service_topology`）的 ORDER BY 需要纳入新维度字段，采用新建表 → 迁移数据 → `EXCHANGE TABLES` 的方式。

## 九、查询示例

```sql
-- 1. 按 TSP 系统 + 品牌 + 环境查看错误率
SELECT
    biz_system,
    biz_brand,
    env,
    count() AS total,
    sum(if(status_code='ERROR',1,0)) AS errors,
    round(errors / total * 100, 2) AS error_rate
FROM platform.tsp_spans
WHERE start_time >= now() - INTERVAL 1 HOUR
GROUP BY biz_system, biz_brand, env
ORDER BY error_rate DESC;

-- 2. 查某个品牌某辆车的完整链路
SELECT * FROM platform.v_trace_detail
WHERE biz_brand = 'bmw'
  AND biz_vin = 'LSVAU2A35N...'
  AND trace_id = 'xxx';

-- 3. 按渠道统计服务调用量
SELECT
    biz_channel,
    service_name,
    count() AS calls
FROM platform.tsp_spans
WHERE biz_system = 'tsp-cloud'
  AND env = 'production'
GROUP BY biz_channel, service_name
ORDER BY calls DESC;

-- 4. 按系统查看服务拓扑
SELECT
    source_service,
    target_service,
    sum(call_count) AS call_count,
    sum(error_count) AS error_count,
    avg(avg_duration_ms) AS avg_duration_ms
FROM platform.tsp_service_topology
WHERE biz_system = 'tsp-cloud'
  AND time >= now() - INTERVAL 3 HOUR
GROUP BY source_service, target_service
ORDER BY call_count DESC;

-- 5. 按环境 + 品牌查看告警事件
SELECT
    env,
    biz_brand,
    severity,
    count() AS alert_count
FROM platform.tsp_alert_events
WHERE alert_time >= now() - INTERVAL 24 HOUR
GROUP BY env, biz_brand, severity
ORDER BY alert_count DESC;
```

## 十、各维度职责总结

| 维度 | 字段 | 谁注入 | 怎么注入 | 透传方式 | 使用场景 |
|------|------|-------|---------|---------|---------|
| 系统 | `biz_system` | Edge Collector | 配置写死 | 不需要透传 | 多 TSP 系统隔离 |
| 环境 | `env` | Edge Collector | 配置写死 | 不需要透传 | 生产/测试/开发隔离 |
| 渠道 | `biz_channel` | APP | SDK Resource | Baggage | 按业务APP渠道分析性能 |
| 品牌 | `biz_brand` | APP | SDK Resource | Baggage | 按品牌查看指标（预留） |
| 租户 | `biz_tenant` | APP | SDK Resource | Baggage | 租户级隔离（预留） |
| VIN | `vin` | APP | SDK Resource | Baggage | 车辆级链路追踪 |
